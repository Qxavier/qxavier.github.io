clojure初探

clojure 主要特性

  • 运行于JVM（CLR/JavaScript）

可以封装为java可调用的jar包，或者去调用java提供的类型

  • 动态类型、强类型语言、动态加载更新代码（本地或远端）

得益于同像性。在不停机情况下，更新程序代码，进行打补丁。

  • 函数式编程语言，函数是一等公民，与其他函数式语言相比，对并发编程提供良好的支持

强制区分引用类型，无需显式编写加解锁过程。

  • Lisp方言，代码即数据，同像性（使用语言本身的数据结构来编写代码）s-表达式，并不是所有的s-表达式可求值，只有可求值的称为form。

大多数高级语言需要通过解释器完成词法分析、语法分析生成AST。并且生成规则由编译器定义，不允许直接操作AST，LISP编写代码的过程就是生成AST的过程。

  • 强大的REPL

在REPL中运行的代码也会编译为java字节码，且是在运行期进行编译，与写在源码文件中再运行没有额外开销。 更多的交互。

(println "helloworld")

(+ 1 3 (+ 1 4))

((fn [x] (+ 1 x)) 3)

(+ 1 (+ 1 ((fn [x] (+ 1 x)) 3)) )

(-> ((fn [x] (+ 1 x)) 3) (+ 1) (+ 1) )

(defn fun [x y] (+ x y) )

数据结构 '(a b :name 19) ;; 列表 [ ‘a ‘b :name 19];; vector {:name “zs” :age 13};; map #{1 2 3};; set

(def v [1 (fn [x] (+ 1 x))]) ((v 1) 3)

clojure数据结构特性 1.依据抽象使用，而非细节，抽象优于实现 100个函数操作1种数据结构比10个函数操作10种数据结构要好。 集合的操作函数基本都是多态的，同样的函数可操作不同的集合类型。 对于java，继承不同的接口，且存在重复实现的情况，代码重用与继承绑定。 例子： (def v [1 2 3]) (conj v 4 5)

(def lst '(1 2 3)) (conj lst 0)

2.不可变且持久的 v lst值并未被改变。对不可变数据进行修改后，产生新的对象，与原对象共享未变化数据的内存。

3.惰性序列（延迟计算） 真正需要集合中的某个元素时，采取调用函数去计算一次，而且仅计算一次，这个过程叫做序列中元素的实例化过程。 例子： (take 5 (drop 2 (cycle [:foo1 :foo2 :foo3]))) (->> [:foo1 :foo2 :foo3] (cycle) (drop 2) (take 5) )

描述无限序列： (defn factorial [n] (apply * (take n (iterate inc 1))))

clojure 并发编程 scala 以消息传递为核心的设计：使用Actor的actor方法，使用不可变对象，不考虑数据共享问题, 使用case来匹配消息并处理。

Scala提供两种Actor：基于线程的Actor和基于事件的Actor。每个基于线程的Actor运行在自己的JVM线程中，它们被Java线程调度器调度，是一个可抢占的优先级调度器。当Actor进入receive代码，线程被阻塞，直到接收到消息。

基于线程的Actor的优势在于，如果Actor内部存在大量计算、或者I/O类阻塞操作，其他Actor可以继续执行。基于线程的Actor模型的缺点在于线程作为Actor的内存和调度开销比较大。当许多Actor开始执行，可能造成内存不足。

基于事件的Actor不会为每个Actor分配线程，它们运行在一个线程之上。基于线程的Actor通过线程阻塞来等待消息，基于事件的Actor通过闭包（closure）。基于事件的Actor是一个轻量级的方法，能够允许非常大数量的Actor存在。但由于所有的Actor在同一个线程中执行，不能保证公平。

为了使用Scala提供的基于事件的Actor，使用react来替代receive即可。

Scala允许代码中混合两种Actor，基于线程和基于事件的Actor可以共存。这为用户提供了在轻量级的基于事件的Actor和并行程度更高的基于线程的Actor之间选择平衡的余地。用户可以根据需要选择合适的Actor。Scala使用线程池来执行Actor，线程池的大小随着需要动态调整。如果只是需要基于事件的Actor，线程池的大小是个常量。当阻塞操作被使用时，比如receive，调度器（单独线程）可能会开始新的Actor线程。调度器周期性地检查任务队列里可执行的任务，它坚持工作者线程是否阻塞，如果需要考试一个新的工作者线程。

clojure STM (software transactional memory) 对引用对象的修改必须在事务内进行。

STM提供了一种显式的锁无关编程模型，这种模型允许多个事务并发地运行，并且在没有发生冲突时所有事务都能毫无滞碍地运行，所以相对其他编程模型而言STM可以提供更好的并发性和线程安全方面的保障。当事务对相同对象或数据的写访问发生冲突时，只有一个事务能够顺利完成，其他事务都会被自动重做。这种重做机制延缓了写操作冲突时竞争失败的那些写者的执行，但却提升了读者和竞争操作的胜利者的执行速度。当对于相同对象的并发写操作不频繁时，其性能就不会受到太大影响。但是随着冲突的增多，程序整体性能将因此变得越来越差。
